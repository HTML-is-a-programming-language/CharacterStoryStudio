# Character Story Studio — 프로젝트 브리핑

타인에이아이 AI-Native Developer 포지션 지원용 메인 포트폴리오 프로젝트의 설계 문서입니다.

## 1. 한 줄 피치

AI 캐릭터와 나눈 대화에서 감정적으로 의미 있는 장면을 추출해, 사용자가 각 단계를 승인·수정하며
30초 내외의 세로형 개인화 모션코믹 영상으로 만드는 서비스.

## 2. 러비더비 / OpenMontage와의 관계

| 구분 | 내용 |
|---|---|
| 러비더비 | 타인에이아이의 AI 캐릭터 채팅 서비스. **핵심 경험(대화·서사)**의 참고 지점. 공개 API가 없어 실제 연동은 하지 않으며, 클론도 아니다. 독자적인 캐릭터/세계관을 가진 자체 데모 채팅 UI를 만든다. |
| OpenMontage | 에이전트 기반 오픈소스 영상 제작 시스템(AGPLv3). **파이프라인 설계(단계별 승인 게이트, 상태 머신, QA 검증)**의 참고 지점. 코드나 도구를 직접 의존성으로 사용하지 않고, 그 구조만 TypeScript로 재구현한다. |

두 서비스를 "합친다"는 것은 코드 수준의 통합이 아니라, **러비더비의 "캐릭터와의 대화"라는 입력**을
**OpenMontage류의 "단계별 승인을 거치는 AI 제작 파이프라인"**에 흘려보내
**"개인화 영상"이라는 출력**을 만드는 컨셉 결합을 의미한다.

## 3. 핵심 사용자 흐름

```
캐릭터 선택
  → AI 캐릭터와 채팅 (데모 채팅 UI)
  → 의미 있는 대화 구간 선택
  → [AI] 대화 분석 → 영상 컨셉 3안 제시        ─┐
  → 사용자 컨셉 선택                            │ 승인 게이트
  → [AI] 5개 씬 스토리보드 생성                 │ (각 단계마다
  → 사용자 스토리보드 승인/수정                 │  진행 전 확인)
  → [AI] 씬별 이미지/대사/연출 생성             │
  → 사용자 씬 승인 또는 부분 재생성              │
  → [AI] 음성·자막·음악 생성 → Remotion 렌더링  │
  → 자동 QA 검증 (해상도/길이/오디오 트랙 등)   ┘
  → 최종 영상 확인 및 다운로드
```

## 4. 차별화 포인트 (포트폴리오 관점)

- **AI 파이프라인 설계 역량**: 단순 텍스트→이미지 생성이 아니라, 상태 머신 기반의 다단계 승인 파이프라인.
- **Human-in-the-loop**: 각 단계에서 사용자가 결과를 보고 승인/재생성/수정할 수 있음. AI 산출물을 그대로 신뢰하지 않는 구조.
- **비동기 처리와 실패 복구**: 생성 작업은 큐 기반으로 처리하고, 재시도/부분 실패/중복 방지를 고려.
- **Provider 추상화**: Chat/Story/Image/TTS 등 외부 AI 호출을 인터페이스로 분리, Mock Provider로 즉시 검증 가능.
- **API 키 없이 동작하는 Demo Mode**: 채용 담당자가 별도 설정 없이 전체 흐름을 체험할 수 있어야 한다.

## 5. MVP 범위

**포함**
- 캐릭터 1~2종(고정 페르소나/외형 설정), 단일 캐릭터 채팅
- 대화 구간 선택 → 분석 → 컨셉 3안 → 스토리보드(5씬) → 씬별 승인/재생성 → 렌더링 → QA → 결과 확인
- Remotion 기반 9:16, 1080×1920, 25~35초 영상 출력
- Demo Mode (Mock Provider + 샘플 데이터로 전체 흐름 체험)

**제외 (추후 로드맵)**
- 다중 캐릭터 동시 등장, 고품질 AI 영상 생성, 얼굴/음성 클로닝
- 결제, 공개 커뮤니티, 캐릭터 마켓플레이스
- 실시간 공동 편집, 모바일 앱
- 실제 러비더비 연동, 유튜브/틱톡 자동 분석
- 벡터DB 기반 장기 기억, 다중 Provider 자동 스코어링

## 6. 기술 스택 제안

- **프론트엔드**: Next.js(App Router), TypeScript strict, Tailwind CSS
- **렌더링**: Remotion
- **백엔드/데이터**: Supabase(Postgres/Auth/Storage), Zod(구조화 출력 검증)
- **비동기 처리**: Redis + BullMQ
- **테스트**: Vitest(unit), Playwright(E2E)
- **패키지 매니저**: pnpm

구조가 필요 이상으로 무거워진다고 판단되면 단순화하고, 그 근거를 남긴다.

## 7. AI 산출물 검증 원칙

- AI가 생성하는 스토리/씬 구조는 스키마(Zod)로 검증한다.
- 씬 길이 합계, 원본 메시지-씬 간 추적 가능성, 캐릭터 설정 일관성을 체크한다.
- 캐릭터 이미지 일관성은 완벽을 보장하지 않되, 승인/재생성 워크플로로 보완한다.

## 8. Demo Mode 요구사항

- API 키 없이 실행 가능해야 한다.
- 샘플 유저·캐릭터·대화·컨셉·스토리보드·이미지·오디오·렌더링 결과를 실제 저장된 데이터로 미리 준비한다(가짜 진행 애니메이션이 아니라 실제 단계별 데이터).

## 9. 구현 순서 (개요)

실제 착수 여부와 각 Phase의 세부 범위는 이 문서를 사용자가 검토한 뒤 별도로 승인받는다.

1. **Phase 0** — 문서화(`docs/PRD.md`, `docs/ARCHITECTURE.md`, `docs/ROADMAP.md`, `docs/DECISIONS.md`) 및 기본 폴더 구조.
2. **Phase 1** — 고정 JSON 데이터를 Remotion으로 MP4까지 렌더링하는 POC (채팅 기능보다 먼저, 렌더링 파이프라인의 실현 가능성을 가장 먼저 검증).
3. **Phase 2 이후** — AI 스토리 생성(Mock→실 Provider), 이미지/음성 생성, 큐 기반 파이프라인, 캐릭터 채팅, 승인 UI, 품질 검증 및 배포 순으로 확장. 각 Phase는 이전 Phase 완료 및 사용자 승인 후 시작한다.

## 10. 열린 질문

- Phase 1 POC에 사용할 구체적인 샘플 스토리(캐릭터, 세계관, 대사)는 아직 정하지 않았다. 실제 코드 착수 전 별도로 논의한다.
- 실제 유료 AI Provider(이미지/영상/음성) 연동 시점과 예산 상한은 추후 결정한다.
